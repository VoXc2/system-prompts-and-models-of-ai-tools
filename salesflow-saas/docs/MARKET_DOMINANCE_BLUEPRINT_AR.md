# Dealix — مخطط هيمنة السوق (Market Dominance Blueprint)

> وثيقة تنفيذية داخلية: تموضع، خندق، GTM، تسعير، تغليف مؤسسي، ثقة/امتثال، سلم قدرات، إسفين تنافسي، خريطة توسع، وتشغيل نجاح العميل.  
> **ملاحظة مصداقية:** أي أرقام أو توقعات مذكورة عن محللين خارجيين (مثل Gartner أو McKinsey) تُستخدم هنا كإطار تفكير؛ **راجع المصدر الرسمي** قبل نشرها في مواد خارجية أو عروض استثمارية.

---

## 0) جملة التموضع القابلة للبيع

**Dealix = Enterprise Growth, Partnerships, CorpDev & Execution OS for GCC companies**

- يربط **القرار → الأدلة → الموافقات → التنفيذ المتين → الأثر المالي → التدقيق**.
- ليس positioning ضعيفًا: «AI CRM» أو «copilot» أو «منصة وكلاء» أو «أداة سير عمل» فقط.

---

## 1) الخندق (Moat design)

| طبقة | ماذا تُبنى | لماذا يصعب الاستبدال |
|------|------------|----------------------|
| حوكمة خارج الـ prompt | سياسات، تفويض دقيق، موافقات، سجلات قرار | المنافس ينسخ واجهة؛ لا ينسخ «التزامًا مؤسسيًا» |
| مخرجات مهيكلة قابلة للتشغيل | حزم JSON/سجلات بصيغة ثابتة (مذكرة، أدلة، مخاطر، موافقة، نية تنفيذ) | التحول من «نص جميل» إلى «أصل تشغيلي» |
| تنفيذ متين (durable) | التزامات أيام/أسابيع مع إعادة محاولة وتعويض وحالة واضحة | الجلسات المحادثية-only تنهار عند الاحتكاك التنظيمي |
| حقيقة تشغيلية + ذاكرة دلالية | Postgres كمصدر حقيقة + استرجاع/تضمين ضمن نطاق المنتج | ربط «ما حدث» بـ «لماذا» و«من وافق» |
| عربي أولاً بمستوى حوكمة | تصنيف، مذكرات، إشعارات، حزم مجلس، مصطلحات، جودة استرجاع | أعمق من RTL أو ترجمة سطحية |
| أسطح إدارة وتدقيق | غرف تنفيذية، مركز موافقات، سجل أدوات/أدلة | يفتح الميزانية العليا ويُقفل المناقصات |

**اختبار الخندق:** هل يصف العميل Dealix دون كلمة «ذكاء اصطناعي»؟ إذا نعم، المسار صحيح.

## 2) حركة GTM (Go-to-market)

### الإسفين الأول (Land)

- **Revenue + Partnerships:** قيمة مالية أوضح، استخدام يومي، بيانات تشغيل غنية (pipeline / شركاء / موافقات).

### ICP مبدئي (قابل للتحديث في `DEALIX_NEXT_LEVEL_MASTER_PLAN_AR.md`)

- B2B معقد في السعودية/الخليج: صفقات طويلة، أطراف متعددة، موافقات متعددة الأقسام، قنوات متعددة.

### ثلاثي المشتري (Buyer triad)

- **CRO / المبيعات:** الإنتاجية وزمن الدورة.
- **CFO / التحكم:** التنبؤ، الهامش، التسعير، الضوابط.
- **CIO / CISO:** الأمن، التفويض، السجلات، حدود الوكالة.

### حزمة Pilot المقترحة

- use case واحد **مقفل بنيويًا**: قرار + حزمة أدلة + موافقة + خطة تنفيذ + تقرير أثر (وليس «تجربة ذكاء» عامة).

## 3) استراتيجية التسعير

**المبدأ:** السعر يتبع **الأثر المالي + نطاق التنفيذ + مستوى الحوكمة + الكيانات التنظيمية**، لا «عدد الرسائل» فقط.

| مستوى | الجمهور | ما يُباع | ما يُقفل تجاريًا |
|--------|---------|----------|------------------|
| Growth Execution | الإيرادات والشراكات | غرفة تنفيذ + موافقات + حزم مهيكلة أساسية | مستخدمون تشغيليون + كيانات نشطة (صفقات/شراكات) |
| Governance & Board | الإدارة العليا / الأمن / القانون | حزم مجلس، سجلات، تفويض دقيق، تقارير تدقيق | SSO، سياسات، احتفاظ، بيئات |
| CorpDev / M&A | CorpDev / استثمار | DD room، risk board، حزم أدلة ثقيلة | مشاريع DD/M&A متزامنة + حدود مستندات |
| Enterprise | مجموعات وهولدنج | multi-entity، فصل بيانات، SLA، تفعيل مخصص | عقد سنوي + خدمات تفعيل |

**إضافات عالية الهامش:** موصلات نظام، حزم تدقيق، اختبارات سطح وكيل/أداة دورية، بيئة معزولة، ضبط مصطلحات عربية قطاعية.

---

## 4) تغليف المؤسسة (Enterprise packaging)

كل عقد Enterprise يجب أن يحدد صراحة:

- **Trust:** سياسات، تفويض، سجلات، إثبات قرار، تصدير تدقيق.
- **Execution:** سير عمل متين، موافقات، تعامل مع الفشل والتعويض.
- **Data:** عقود أحداث، جودة بيانات، قاموس مقاييس، نسب (lineage) أساسي.
- **Security:** حدود وكالة، مرونة ضد حقن التوجيه، ضوابط مخرجات حساسة، قواعد تجميد إصدار عند المخاطر.
- **Success:** cadence أسبوعي/شهري، قياس ROI، مجلس نجاح ربعي.

---

## 5) نقاط بيع الثقة والامتثال (Trust / compliance)

- **PDPL-by-design:** دورة حياة بيانات واضحة (جمع، تخزين، بحث، مشاركة، تحليل، حذف) مع تصنيف وحقوق وموافقات وسجلات — انظر سياسات المنتج في `docs/legal/` وطبقة `services/pdpl`.
- **جاهزية أمنية محلية:** عند العملاء الذين يشيرون إلى **NCA ECC** كإطار توقعات، تُعرض كـ «جاهزية تشغيل وتصميم» وليست شهادة جاهزة دون تقييم بيئة العميل.
- **Evidence-first:** لا توصية حرجة بلا حزمة أدلة قابلة للمراجعة.
- **Least privilege + حدود الوكيل:** التنفيذ الخارجي يمر عبر workflow + policy + approval.

## 6) سلم القدرات (Capability ladder)

| مستوى | الوصف |
|-------|--------|
| L0 | مساعدة تنفيذية: تحضير ومسودات وتنسيق |
| L1 | توصيات محكومة: سياسات + موافقات بشرية + سجلات |
| L2 | عمليات شبه آلية: مهام متكررة داخل حدود + سجل أدوات |
| L3 | OS跨وظيفي: إيرادات + شراكات + قانون + مالية + PMO على مسار قرار واحد |
| L4 | محفظة: CorpDev / M&A / توسع / PMI بنفس الحوكمة |
| L5 | Board-grade: حزم مجلس، مخاطر مؤسسية، مؤشرات موحدة |

**قاعدة:** لا ترفع الاستقلالية قبل تثبيت L1 في الإنتاج.

## 7) الإسفين التنافسي (Competitive wedge)

**الرسالة:** «نُقصّر زمن القرار ونرفع جودة التنفيذ مع حوكمة قابلة للتدقيق» في **Revenue + Partnerships** أولًا.

- مقابل CRM + وكلاء: نحن **التزام وتنفيذ وأثر** لا تحسين إدخال فقط.
- مقابل منصات وكلاء: **وكلاء بلا حوكمة = تكلفة مخاطرة**.
- مقابل أدوات سير عمل: **سير عمل بلا أدلة وROI = لوحة معلومات**.

|---------|----------|

## 11) مخرجات مقفلة بنيويًا (Structured operational artifacts)

أي قرار حرج يجب أن يُخزَّن ويُصدَّر كحزمة مهيكلة (مثال أسماء حقول/وثائق):

- `memo_json`
- `evidence_pack_json`
- `risk_register_json`
- `approval_packet_json`
- `execution_intent_json`

**ملاحظة تقنية:** الالتزام بـ schema (مثل مخرجات مهيكّرة من مزودي LLM) أقوى من «JSON حر» فقط — يُربط في التصميم بـ Pydantic في الـ backend وعقود API.

## 12) طائرتان: قرار وتنفيذ

- **Decision plane:** تفكير، تحليل، إعداد حزم، مقارنة سيناريوهات، مراجعة بشرية (HITL).
- **Execution plane:** موافقات، DD، توقيع، إطلاق، PMI — يجب أن **تصمد** أيامًا وتعيد المحاولة وتتحمل الأعطال.

**توجيه معماري:** منطق تفكير حالة (stateful reasoning) يختلف عن **workflow runtime** للالتزامات الطويلة؛ عند التوسع المؤسسي، قيّم مكوّنًا للتنفيذ المتين يتناسب مع متطلبات الـ retries والتعويض (مثل أنماط العمل المؤجلة/الدائمة في الخلفية الحالية + أي طبقة مستقبلية للتنسيق الطويل).

## 16) حلقات هندسية مقترحة (Epics) مرتبطة بالمخطط

1. **حزم مهيكرة في API:** نماذج Pydantic + تخزين + تصدير لـ memo/evidence/risk/approval/execution intent.  
2. **Approval Center:** حالة موافقة، أدوار، سجل تدقيق، ربط بكيانات الصفقة/الشريك.  
3. **مراقبة:** OpenTelemetry أو ما يعادله على مسار الطلبات والوكلاء مع ربط trace وlog.  
4. **تفويض دقيق:** نموذج صلاحيات يتوسع من JWT/أدوار حاليّة إلى علاقات مؤسسية (حسب الطلب).  
5. **واجهات تنفيذية:** صفحات أو مسارات UI للأسطح الثمانية بشكل تدريجي (حتى لو MVP لكل سطح).
