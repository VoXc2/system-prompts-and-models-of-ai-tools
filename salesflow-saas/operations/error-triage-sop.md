# إجراءات معالجة الأخطاء - Dealix

## مستويات الخطورة

| المستوى | الوصف | وقت الاستجابة | أمثلة |
|---------|-------|-------------|--------|
| P1 - حرج | النظام متوقف كلياً | < 1 ساعة | الموقع لا يعمل، قاعدة البيانات معطلة |
| P2 - عالي | ميزة رئيسية معطلة | < 4 ساعات | واتساب لا يرسل، الدخول لا يعمل |
| P3 - متوسط | ميزة ثانوية معطلة | < 24 ساعة | تقرير لا يحمّل، خطأ في العرض |
| P4 - منخفض | مشكلة تجميلية | < 48 ساعة | خطأ إملائي، تنسيق غير مثالي |

## خطوات المعالجة

### 1. التشخيص (5-15 دقيقة)
- تحقق من Docker containers: `docker ps`
- تحقق من logs: `docker logs <container>`
- تحقق من health endpoint: `curl /api/v1/health`
- تحقق من اتصال قاعدة البيانات
- تحقق من Redis

### 2. الحل السريع
- إعادة تشغيل Container المتأثر: `docker restart <name>`
- إذا لم ينجح: `docker-compose down && docker-compose up -d`
- إذا مشكلة بيانات: تحقق من PostgreSQL مباشرة
- إذا مشكلة ذاكرة: `docker system prune` + restart

### 3. التصعيد
- P1: إبلاغ المؤسس فوراً + بدء الإصلاح
- P2: إبلاغ المدير التقني + تحديث حالة كل ساعة
- P3/P4: تسجيل في قائمة المهام + إصلاح بالدور

### 4. التراجع (Rollback)
إذا التحديث الأخير سبب المشكلة:
```bash
git log --oneline -5  # تحديد الإصدار السابق
git revert HEAD       # التراجع عن آخر تغيير
docker-compose build && docker-compose up -d
```

### 5. ما بعد الحادثة (Post-mortem)
- ما الذي حدث؟
- متى اكتُشف؟
- كيف تم الإصلاح؟
- ما السبب الجذري؟
- كيف نمنع تكراره؟
- هل يحتاج تحديث الـ monitoring؟
